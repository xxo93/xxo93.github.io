B+ 树结构
B+ 树是 MySQL InnoDB 存储引擎的默认索引结构。它是一种多路平衡搜索树，所有数据都存储在叶子节点，叶子节点之间通过链表相连。
🌳

多路平衡

每个节点可有多个子节点（不只是二叉），树的高度很低，通常 3~4 层就能存储上千万条记录。

📄

数据在叶子节点

内部节点只存索引键和指针，所有实际数据都在叶子节点中，查询路径长度一致。

🔗

叶子节点链表

所有叶子节点通过指针相连，支持高效的范围查询（如 BETWEEN、ORDER BY）。

⚡

查找效率 O(log n)

对于 1500 万条记录，只需 3~4 次磁盘 I/O 即可定位数据，性能极佳。

B+ 树 vs B 树
B+ 树（MySQL 使用）
数据只在叶子节点 → 内部节点更小 → 每页能存更多索引键 → 树更矮 → I/O 更少。叶子链表支持范围扫描。
B 树
数据在所有节点 → 内部节点更大 → 树更高 → 更多 I/O。不支持高效范围查询。
聚簇索引（Clustered Index）
聚簇索引将数据行直接存储在 B+ 树的叶子节点中。InnoDB 的主键索引就是聚簇索引——找到主键，就直接拿到了完整行数据，无需额外查找。

二级索引 & 回表
二级索引的叶子节点不存储完整行数据，只存储（索引列值 → 主键值）。通过二级索引查到主键后，需要再回到聚簇索引查找完整数据，这个过程叫做回表。

覆盖索引（Covering Index）
当查询所需的所有字段都包含在索引中时，MySQL 无需回表，直接从索引就能返回结果。这就是覆盖索引——索引"覆盖"了查询需求。
✗ 需要回表 · SELECT *
SELECT * FROM users WHERE name = 'Ivy'

联合索引 & 最左前缀原则
联合索引 (dept, age) 按照字段顺序排序：先按 dept 排序，dept 相同时再按 age 排序。查询时必须从最左边的字段开始匹配，才能有效利用索引。

索引 (a, b, c)
✓ WHERE a = 1
✓ WHERE a = 1 AND b = 2
✓ WHERE a = 1 AND b = 2 AND c = 3
⚠ WHERE a = 1 AND c = 3（只用到 a）
✗ WHERE b = 2（跳过了 a）
✗ WHERE c = 3（跳过了 a, b）
✗ WHERE b = 2 AND c = 3（跳过了 a）
